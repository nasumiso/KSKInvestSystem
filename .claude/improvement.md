# 設計・アーキテクチャレベルの改善点

このドキュメントは、株式市場分析システムの設計とアーキテクチャにおける改善提案をまとめたものです。

## 1. **データ永続化層の改善**

**現状の問題:**
- Pickleファイル (`stocks.pickle`) による単一ファイルデータベース
  - 1件更新でも全銘柄（数千件）を読み書き必要
- データ破損リスク、並行アクセス制御なし
- スキーマ管理が不明確（辞書構造の暗黙的な契約）

**段階的移行アプローチ:**
1. **ステップ1（最優先）: pickle → shelve**
   - Python標準ライブラリの永続的辞書（追加依存なし）
   - 部分的な読み書きが可能（全体のメモリ展開不要）
   - 移行コスト低、インターフェースはほぼ同じ

2. **ステップ2（中期）: shelve → SQLite**
   - クエリによる高速検索・フィルタリング
   - ORM (SQLAlchemy等) によるデータモデルの明示化
   - 移行コスト中〜高だが、分析要件に最適

3. **ステップ3（必要時）: SQLite → PostgreSQL**
   - マルチユーザー対応、大規模データ向け

## 2. **レイヤー分離の欠如**

**現状の問題:**
- データ取得（スクレイピング）、ビジネスロジック（分析・ランキング）、データ永続化が各モジュール内で混在
- 例: `price.py`がYahoo/Kabutanからのスクレイピングと技術指標計算の両方を担当

**改善案:**
```
layers/
├── data_sources/      # スクレイピング専用（Yahoo, Kabutan, etc）
├── repositories/      # データアクセス層（DB操作の抽象化）
├── domain/           # ビジネスロジック（指標計算、ランキング）
├── services/         # ユースケース層（分析フロー制御）
└── presentation/     # CSV出力、レポート生成
```

## 3. **依存関係の循環と密結合**

**現状の問題:**
- `make_stock_db.py`が`price.py`, `gyoseki.py`, `shihyou.py`等を直接インポート
- 各モジュールが`ks_util.py`の多数の関数に依存
- モジュール間の責任範囲が不明確

**改善案:**
- 依存性注入（DI）パターンの導入
- インターフェース/抽象基底クラスによる疎結合化
- `ks_util.py`を機能別モジュールに分割（`http_client.py`, `cache_manager.py`, `logger.py`等）

## 4. **キャッシュ戦略の一元化不足**

**現状の問題:**
- UPD_*定数による4段階制御は存在するが、実装が各モジュールに分散
- キャッシュの有効期限ロジックが関数ごとにハードコード（例: `INTERVAL_DAY = 7`）
- ファイルキャッシュとメモリキャッシュ（pickle DB）の整合性管理が不明確

**改善案:**
- キャッシュマネージャーの導入（TTL、無効化ポリシーを一元管理）
- デコレーターパターンでキャッシュロジックを分離
```python
@cache_with_ttl(days=7, invalidate_on_earnings=True)
def get_stock_master_data(code_s):
    ...
```

## 5. **エラーハンドリングとリトライ戦略**

**現状の問題:**
- スクレイピング失敗時の処理が統一されていない
- HTMLフォーマット変更への対応が各パーサー内にハードコード
- セマフォによるレート制限はあるが、429エラー等への動的対応がない

**改善案:**
- スクレイパー基底クラスでリトライ・フォールバック機能を実装
- HTMLパーサーの戦略パターン（複数バージョンのパーサーを試行）
- Circuit Breakerパターンでサービス障害時の連鎖防止

## 6. **設定管理の外部化**

**現状の問題:**
- 定数が各モジュールにハードコード（例: `INTERVAL_DAY = 7`, `MAX_REQUESTS = 3`）
- データディレクトリパスが`ks_util.py`で固定

**改善案:**
```
config/
├── default.yaml      # デフォルト設定
├── production.yaml   # 本番環境設定
└── development.yaml  # 開発環境設定
```
- 環境変数による設定切り替え
- Pydantic等による設定値のバリデーション

## 7. **非同期処理の改善**

**現状の問題:**
- `ThreadPoolExecutor`による並列化は実装済みだが、スレッド数（5）が固定
- セマフォ制限（3）とスレッド数（5）の関係が非自明
- I/O待機が多いスクレイピングでasyncioが未活用

**改善案:**
- `asyncio` + `aiohttp`への移行検討（より効率的なI/O多重化）
- 動的なワーカー数調整（スクレイピング対象数に応じた最適化）

## 8. **テスタビリティの向上**

**現状の問題:**
- HTTPリクエストが実際の外部サービスに依存（モック化が困難）
- ユニットテストが`test_functional.py`のみで統合テストが主体
- 副作用を持つ関数（DBへの直接書き込み）が多い

**改善案:**
- リポジトリパターンでデータアクセスを抽象化
- HTTPクライアントをインターフェース化（`requests_mock`や`vcr.py`の活用）
- 純粋関数と副作用の分離（関数型プログラミングの原則）

## 9. **スケジューリングとワークフロー管理**

**現状の問題:**
- `shintakane_cron.sh`による単純なcron実行
- 依存関係のあるタスク（市場データ更新→銘柄分析）の順序制御がシェルスクリプト頼み
- 失敗時のリトライや通知機能なし

**改善案:**
- Airflow、Luigi、Prefect等のワークフローエンジン導入
- タスク依存関係の明示的な定義
- 実行履歴、ログ、アラートの統合管理

## 10. **データバリデーション**

**現状の問題:**
- スクレイピングしたデータの妥当性チェックが不十分
- 型ヒントは部分的（コード全体に一貫性なし）
- 株価が負数、出来高がNaN等の異常データへの対処が不明確

**改善案:**
- Pydanticによるデータモデル定義と自動バリデーション
```python
from pydantic import BaseModel, validator

class StockPrice(BaseModel):
    code_s: str
    price: int
    volume: float

    @validator('price')
    def price_must_be_positive(cls, v):
        if v <= 0:
            raise ValueError('price must be positive')
        return v
```
